Abstract

Prior works on training software engineering agents have explored utilizing existing resources such as issues on GitHub repositories. However, these approaches face limitations: reliance on static repositories limits flexibility, and a focus on issue resolution restricts versatile skill learning.

To overcome these challenges, we introduce SWE-Playground, a novel data generation pipeline designed to support the training of more versatile coding agents. Unlike prior efforts, SWE-Playground synthetically generates projects and tasks from scratch with strong language models and agents, eliminating reliance on external data sources. This allows us to tackle a much wider variety of coding tasks, such as reproducing issues by generating unit tests and implementing libraries from scratch. Results indicate that SWE-Playground is highly data efficient, enabling agents to reach comparable performance with significantly fewer trajectories than previous work.

How It Works

Main Data Generation Pipeline

1. Project Proposal

An LLM generates a comprehensive proposal outlining scope, objectives, and constraints (e.g., specific domains or languages) to ensure diversity.

2. Task Proposal

The project is decomposed into step-by-step tasks. A detailed checklist is generated, explicitly specifying the requisite unit tests for each task.

3. Repository Setup

An agent establishes the foundational code structure, file paths, and environment (Docker/dependencies) without implementing core functionality.

4. Unit Test Generation

An active agent generates strict unit tests based on the checklist. It executes the tests to verify imports and validity before implementation begins.

5. Functionality Implementation

An agent implements functionality using the tests as a guide (TDD). A mutual verification loop ensures correctness before the final trajectory is saved.

Task Specific Generation

Issue Resolution

Simulating SWE-bench: Based on the fully functional repository generated by our pipeline, we use an agent to inject a specific bug. The existing test suite is used to verify the bug injection.

Issue Reproduction

Simulating SWT-Bench: We inject a bug but hide it from the test suite. The agent is tasked with writing a new test script that successfully exposes the faulty behavior, rather than fixing the code itself.

Library Generation

Simulating Commit-0: We take a completed repository and replace all function bodies with pass statements. The agent must re-implement the entire library from scratch, using the structure as a reference.

Key Findings

Versatility Across Benchmarks: Our agents outperform baselines on SWT-Bench (Issue Reproduction) and Commit-0 (Library Generation from Scratch), while maintaining competitive performance on SWE-bench.
Superior Data Efficiency: Using only 704 trajectories, our SWE-Play-mix model rivals baselines like R2E-Gym and SWE-smith which use over 2,000 trajectories.

Impact of Trajectory Composition

To investigate how different trajectory types influence model performance, we conduct an ablation study by training two additional 7B models: SWE-Play-general (280 general-purpose trajectories) and SWE-Play-swe (213 SWE-bench specific trajectories).

Results indicate that training solely on general trajectories yields performance gains on three of the four metrics. The only exception is a nearly identical coverage delta on SWT-Bench. This demonstrates that generalist training effectively develops core coding abilities. We attribute this to the intrinsic design of the SWE-Playground pipeline: models build projects from blank templates (analogous to Commit-0) and naturally resolve bugs during implementation (analogous to SWE-bench). Furthermore, explicitly prompting the model to write test scripts empowers it with unit test generation skills.

Investigation into Data Efficiency

Training time scaling trends of SWE-Gym and the projected trend of SWE-Playground. Our projection suggests that our model could achieve nearly 45% resolved rate with equivalent data volume.

Our model demonstrates remarkable data efficiency. SWE-Play-mix is only outperformed by R2E-Gym on SWE-bench Verified, despite R2E-Gym using a dataset nearly five times larger (3,321 vs 704). Furthermore, our SWE-Play-swe-7B surpasses SWE-Gym-7B while using less than half the volume of training trajectories.

To understand this efficiency, we analyzed the trajectory statistics (Table 4). Our trajectories contain 2-3x more tokens and messages on average compared to baselines. Additionally, the significantly higher proportion of bash execution actions (41.7%) indicates that our agents learn a robust, iterative development methodology.

Future Work

Broader Coding Capaiblities

Incorporating a broader range of coding capabilities and benchmarks, such as SWE-bench Multimodal and SWE-perf, would further validate the extensibility of our framework. The features inherent to such benchmarks are highly compatible with the generative capabilities of our pipeline, enabling the testing of diverse software engineering skills.

Reinforcement Learning

Conducting Reinforcement Learning (RL) experiments is a critical next step. Our pipeline creates a scenario where unit test generation and code implementation mutually verify each other. Training an agent to master both skills through Reinforcement Learning will allow us to explore the potential for developing self-verifying and self-improving coding agents.
